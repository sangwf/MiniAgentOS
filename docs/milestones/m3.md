# Milestone 3: Native Agent Core

## Objective

Turn MiniAgentOS from a runtime with a constrained natural-language shell into
an actual single-agent system that can accept ordinary goal text, use a model
to interpret the goal, compile that interpretation into a structured Intent IR,
choose bounded capabilities, execute them, and return a result directly to the
user.

M3 is the milestone where `Goal >` stops being "natural-language-shaped command
input" and starts becoming a real agent interface.

## Why M3 Exists

M0 proved MiniAgentOS can boot, act, and be evaluated.

M1 proved MiniAgentOS can execute a governed skill runtime with policy and
real-model validation.

M2 proved MiniAgentOS can accept supported goal text directly at `Goal >`
without delegating goal compilation to a host-side translator.

What M2 does **not** prove is the thing the product actually wants:

- ordinary natural-language interaction
- model-driven task understanding
- task completion without forcing the user into a rigid goal template

M3 exists to prove these stronger claims:

- MiniAgentOS can treat free-form user text as the input to an agent loop
- goal understanding lives in a runtime-owned agent core, not in a regex-style
  shell compiler
- goal understanding is compiled into a stable task-shaped IR rather than
  hidden in ad hoc prompt inheritance between model calls
- the default manual path does not depend on a host-side interpretation service
- the official live path does not depend on a host-side model gateway either
- manual QEMU interaction and harness automation exercise the same agent core
- the runtime can complete a useful task and return the result directly to the
  user

## Product Meaning

M3 is where MiniAgentOS becomes an actual agent runtime.

By the end of M3, the right mental model should no longer be:

"a kernel demo with some harness-driven task paths"

It should be:

"a bootable single-agent runtime whose main interface is natural-language task
execution"

## Non-goals

- multi-agent orchestration
- unrestricted autonomous browsing
- full desktop automation
- long-term memory or retrieval systems
- self-modifying code flows
- unrestricted tool growth
- replacing harness evaluation with manual demos

## Core Definition

MiniAgentOS M3 is done when the runtime supports:

- one model-driven agent loop as the primary `Goal >` path
- one runtime-owned intent compiler that emits a structured Intent IR
- one bounded capability registry used by that loop
- one result path that returns directly to the shell by default
- one preserved structured path for harness and low-level debugging
- one harness suite that evaluates natural-language task completion rather than
  template matching alone
- zero required host-side interpretation endpoints for the default manual path
- zero required host-side model-provider gateways for the official live path

## Agent Standard

MiniAgentOS should treat "agent" to mean all of the following:

- the user can type ordinary natural language at `Goal >`
- the runtime uses a model to interpret the request
- the runtime turns the interpreted request into a structured Intent IR and a
  bounded execution plan
- the runtime selects capabilities from a small governed registry
- the runtime completes the task or refuses it with a meaningful reason
- the runtime returns a result directly to the terminal by default
- the runtime can do that without requiring a host-side `/interpret` service
- the runtime can do that on the official live path without routing model
  traffic through a host-side OpenAI gateway
- if `OPENAI_API_KEY` has not been injected yet, the runtime prompts the user to
  provide it instead of silently falling back to a host interpretation service
- all important actions remain visible in trace

If one of those properties is missing, the system may still be useful, but it
does not yet satisfy the intended M3 bar.

## Architecture Direction

M3 should align MiniAgentOS to a simple agent architecture inspired by the pi
framework family:

`goal shell -> agent core -> capability registry -> primitives`

### Required Internal Layers

M3 should leave the runtime with at least these conceptual layers:

- `goal_shell`
  human-facing input and output at `Goal >`
- `agent_core`
  the loop that sends goal/context to the model, receives structured intent,
  compiles Intent IR, and advances execution
- `intent_ir`
  the normalized, policy-ready task representation produced by the agent core
- `model_adapter`
  provider-facing request/response logic and error handling
- `capability_runtime`
  the bounded execution surface for fetch, summarize, post, state access, and
  later actions

The agent core is the key requirement. It must become the main path for manual
interaction instead of the current narrow local compiler.

## Intent Understanding Requirement

M3 should treat goal understanding as a compilation step, not as a loose prompt
router.

The runtime should first classify the interaction, then normalize it into a
stable Intent IR that later phases can execute and evaluate.

The minimum Intent IR should carry fields equivalent to:

- `interaction_kind`
- `action`
- `source_url`
- `sink_url`
- `max_items`
- `output_language`
- `style`
- `requires_clarification`
- `reason`

For example, a request such as:

```text
Summarize https://example.com in three bullet points in Chinese.
```

should compile into a representation equivalent to:

```json
{
  "status": "ok",
  "interaction_kind": "task",
  "action": "summarize_url",
  "source_url": "https://example.com",
  "sink_url": null,
  "max_items": 3,
  "output_language": "zh",
  "style": "bullet",
  "requires_clarification": false,
  "reason": null
}
```

The important design rule is that language, style, and similar user-visible
constraints must not survive only as raw text copied into later prompts. They
must become explicit runtime state.

### Explicit Dependency Boundary

The official M3 bar forbids a manual interaction dependency on a host-side
goal-interpretation endpoint such as `http://10.0.2.2:8084/interpret`.

Host-side services may still exist for harness fixtures, evaluation sinks, or
mock-only model fixtures, but they must not be the place where the default
manual `Goal >` path gets its essential goal understanding from.

### Provider Boundary

The official M3 live path also forbids a host-side model gateway from standing
in for MiniAgentOS at the provider boundary.

That means the runtime itself must own:

- the HTTPS request to `api.openai.com`
- the `Authorization` header using `OPENAI_API_KEY`
- the request/response contract for the OpenAI model API
- the interpretation of the returned model output

The host may still provide deterministic mock fixtures for regression or
prototype bridging flows, but those do not satisfy the official live M3 bar.

### Transport Proxy Exception

MiniAgentOS may still rely on a plain host-side transport proxy when the
developer environment requires proxying for outbound internet access.

That is acceptable because it does not move goal understanding or provider
ownership out of MiniAgentOS. It only forwards network traffic.

The important distinction is:

- allowed: a dumb host-side transport proxy used for HTTPS egress
- forbidden: a host-side `/interpret` service
- forbidden: a host-side OpenAI application gateway that owns request shaping
  or result parsing for the official live path

The current runtime is expected to use `10.0.2.2:7897` for this transport
proxy when proxying is required.

## Capability Model

M3 should keep the capability surface intentionally small.

The runtime does **not** need a giant toolset. It needs a small number of
clear, inspectable capabilities that the agent can use reliably.

The default M3 capability set should stay close to:

- `fetch_url`
- `summarize_text`
- `call_model`
- `render_result`
- `post_result`

### Capability Rules

- capabilities must be planner-visible
- capabilities must remain policy-controlled
- capability invocation must remain trace-visible
- capability outputs should support both agent-facing structure and
  user-facing rendering

## Input Model

M3 keeps two entry paths, but only one is primary.

### Primary Path

Free-form natural language typed at `Goal >`

Examples:

```text
Summarize https://example.com for me.
```

```text
Read this page and give me three key takeaways.
```

```text
Fetch https://example.com and post a summary to http://10.0.2.2:8081/result.
```

These should route into the same agent core.

The first and second examples above must work in manual QEMU interaction
without pre-starting a host-side interpretation gateway.

### Secondary Path

Structured task input for harness and debugging.

This path stays because it is useful for deterministic evaluation, but it is no
longer the product-defining interface.

## Execution Model

The M3 loop should look like this:

`goal text -> interaction classification -> model interpretation -> Intent IR -> bounded action/plan -> policy check -> capability call -> result -> next step or finish`

This does **not** require open-ended chain-of-thought dumping or unrestricted
autonomy. It only requires that the runtime use a model to interpret and route
the task instead of relying on hard-coded pattern matching as the primary
mechanism.

For the default manual path, that interpretation must be owned by MiniAgentOS
itself rather than delegated to a host endpoint.

## Default M3 Task Bar

The first official M3 success path should be:

- user enters a normal natural-language request to summarize one URL
- MiniAgentOS interprets it through the model-backed agent core
- MiniAgentOS performs one real fetch
- MiniAgentOS returns the summary directly to UART
- no result sink is required for the default manual path

Example:

```text
Summarize https://example.com in three bullet points.
```

This is the default product bar because it is the smallest task that feels like
a real agent instead of a templated shell.

## Governed External-Effect Path

M3 must also preserve one governed task family with an external effect:

- fetch one source URL
- summarize it
- post the result to one sink URL

That path exists to prove the agent core can still drive the existing governed
M1 capability runtime rather than bypassing it.

## Model Requirement

For live model-backed paths, the required provider remains:

- `OPENAI_API_KEY`
- `gpt-5-mini`

M3 must not treat model use as a host-side trick. The runtime-owned agent core
must meaningfully depend on model interpretation for the manual natural-language
path, and the runtime must call the OpenAI API directly from MiniAgentOS on the
official live path.

This means the official M3 implementation must include a real in-guest model
adapter for OpenAI. A host-side interpretation gateway does not satisfy the
official M3 bar, and neither does a host-side OpenAI proxy.

### Secret Handling Requirement

M3 also requires an explicit way to supply `OPENAI_API_KEY` to MiniAgentOS for
manual and harness-driven runs.

The exact mechanism is still an implementation choice, but it must be owned by
the runtime contract, for example:

- a boot-time secret injection path
- a controlled serial secret command
- a harness-provided secret channel

What does **not** count is keeping the key only on the host and asking a host
gateway to spend it on the runtime's behalf.

## Observability

M3 must preserve and extend traceability.

At minimum, trace should make these stages visible:

- goal reception
- interaction classification
- model interpretation start
- model interpretation result
- intent compilation
- plan or action selection
- policy checks
- capability calls
- capability results
- terminal status

### Minimum Trace Vocabulary

- `goal_received`
- `goal_text_received`
- `agent_loop_started`
- `interaction_classified`
- `model_interpretation_started`
- `model_interpretation_result`
- `intent_compiled`
- `clarification_required`
- `plan_created`
- `policy_checked`
- `skill_called`
- `skill_result`
- `goal_completed`
- `goal_refused`
- `goal_failed`

M3 may preserve some M2 event names for continuity, but the important shift is
that the trace should reveal a real model-backed interpretation step.

## Error Handling

M3 should introduce explicit failure classes for agent-facing model use.

At minimum, the runtime or harness-visible result path should distinguish:

- unsupported goal
- unsupported conversation or chat-only request
- policy refusal
- fetch failure
- provider/auth failure
- provider/rate-limit or quota failure
- provider/timeout failure
- malformed interpretation result

This is important because once the agent core is model-driven, "failed" is too
coarse to debug or evaluate usefully.

## Harness Role

The harness remains first-class in M3, but its role is:

- feed user-like natural-language tasks
- provide controlled fixtures and sinks
- validate trace and terminal outputs
- validate governed external effects when applicable
- help inject secrets or runtime configuration when needed

The harness should not remain the place where MiniAgentOS gets its essential
goal understanding from.

If the harness must start an interpretation gateway for a run to succeed, that
run may still be useful as a bridge prototype, but it does not count as the
official M3 success path.

Likewise, if the harness must start a host-side OpenAI gateway for a live run
to succeed, that run is still useful as prototype coverage, but it does not
count as the official M3 live path.

## M3 Harness Suite

The minimum M3 suite should include at least these cases:

### `m3-nl-summarize-url`

- free-form natural language request
- one source URL
- no sink requirement
- expected result: summary appears in UART
- expected result: `goal_completed`

### `m3-nl-summarize-url-openai`

- same task family
- live provider path
- uses `OPENAI_API_KEY` and `gpt-5-mini`
- MiniAgentOS connects to OpenAI directly from the guest
- expected result: `goal_completed`

### `m3-nl-post-summary`

- free-form natural language request with external effect
- source URL and sink URL
- expected result: sink receives result
- expected result: `goal_completed`

### `m3-nl-policy-refusal`

- natural-language request asks for a disallowed target or capability
- expected result: `goal_refused`

### `m3-nl-interpretation-error`

- model result is malformed, unavailable, or denied
- expected result: `goal_failed`
- trace must make the interpretation failure explicit

## Definition Of Done

Milestone 3 is done only when the real MiniAgentOS runtime satisfies all of the
following:

### Product Bar

- `Goal >` accepts normal natural-language requests, not only templated command
  shapes
- the default summarize-a-URL task can be completed directly from QEMU manual
  interaction
- the result is returned directly to the shell without requiring a sink
- language and style constraints such as `in Chinese` or `three bullet points`
  survive as explicit structured intent, not only as copied raw goal text
- the default summarize-a-URL task does not require a host-side interpretation
  service such as `10.0.2.2:8084`
- the official live summarize-a-URL task does not require a host-side OpenAI
  gateway
- the official live summarize-a-URL task may use a plain host-side transport
  proxy such as `10.0.2.2:7897`, but MiniAgentOS still owns the provider
  request, headers, and response parsing end to end

### Agent Core

- the primary manual path uses a model-driven agent core
- the agent core compiles a structured Intent IR before capability execution
- the Intent IR explicitly owns at least:
  - interaction kind
  - action
  - source URL
  - sink URL when present
  - output count/style constraints
  - output language when requested
- the agent core selects bounded capabilities rather than directly executing one
  hard-coded path
- the current narrow deterministic M2 compiler is either removed from the main
  path or downgraded to explicit fallback behavior
- any bridge implementation that still depends on a host `/interpret` endpoint
  is treated as pre-M3 prototype work, not milestone completion
- any bridge implementation that still depends on a host OpenAI gateway is
  treated as pre-M3 prototype work, not milestone completion

### Provider Ownership

- MiniAgentOS issues the live OpenAI HTTPS request itself
- MiniAgentOS owns the live request/response schema for `gpt-5-mini`
- MiniAgentOS has an explicit secret-injection path for `OPENAI_API_KEY`

### Capability Governance

- capability selection remains policy-bound
- external-effect tasks still go through the governed capability runtime
- the agent core does not bypass policy to complete tasks

### Observability

- model interpretation is visible in trace
- intent compilation is visible in trace
- terminal success, refusal, and failure remain machine-checkable
- harness artifacts remain diffable under `output/`

### Harness

- the repository contains a dedicated M3 suite
- one deterministic regression bar passes
- one live OpenAI-backed M3 case passes through a guest-direct OpenAI path
- the M1 suite still passes after the agent-core path lands

### Engineering Bar

- M3 logic does not collapse into one large shell handler
- the runtime is clearly split into:
  - goal shell
  - agent core
  - intent IR
  - model adapter
  - capability runtime

## Final Meaning

- M3 proves MiniAgentOS can actually behave like an agent
- M3 proves natural language is the main interface, not a cosmetic wrapper
- M3 proves the runtime owns goal understanding rather than outsourcing it to
  harness glue
- M3 proves goal understanding is explicit runtime state rather than a hidden
  side effect of prompt wording
- M3 proves the runtime owns its live model connection rather than outsourcing
  provider access to a host gateway
