# Milestone 7: Durable Memory And Context Runtime

## Status

Partially implemented.

M7 now has:

- a first fixture-backed memory suite in the harness
- first QEMU-backed live memory-inspection and truthful-compaction slices
- a first real in-guest memory inspection slice
- a first real in-guest truthful-compaction slice

The full explicit working-memory runtime is still not complete in the real
guest, but the artifact contract, fixture substrate, evaluator surface, first
deterministic and first live M7 cases, bounded guest memory store, prompt
memory sections, inspection commands, and first automatic guest compaction
policy are now in place.

## Objective

Add the first explicit memory and context substrate to MiniAgentOS so the
runtime can:

- retain the right facts across long coding and research loops
- compact large tool results truthfully instead of blindly appending them
- expose what is currently remembered and why
- resume interrupted work without restarting from an empty prompt

M7 is where MiniAgentOS should move from:

"a runtime that can do bounded tasks in one session"

to:

"a runtime that can carry bounded, truthful working memory across long tasks
and restarts."

## Why M7 Exists

M6 gives MiniAgentOS a real search and research loop. M5 gives it a real coding
loop. But the runtime still assembles context in a mostly direct, turn-by-turn
way:

- `Current request`
- `Latest tool result`
- `Known sources`
- `Session state`
- `Recent conversation`

That is enough for short tasks. It is not enough for long ones.

Without explicit memory management, MiniAgentOS will eventually hit the same
failure modes that affect many agent systems:

- large tool results dominate the prompt
- old turns linger without being useful
- important facts are lost when context is trimmed
- follow-up turns become harder to audit
- interrupted tasks are expensive to resume

M7 exists to make memory and context a first-class runtime concern instead of a
side effect of concatenating prior turns.

## Product Meaning

By the end of M7, the right mental model should be:

"MiniAgentOS is a bounded agent runtime with explicit, inspectable, durable
working memory."

This does not require MiniAgentOS to become a general database, a universal
knowledge graph, or an unbounded chat archive. It does require the runtime to
own:

- what it remembers
- what it forgets
- what it compresses
- what it carries into the next turn

## Implementation Principle

M7 should follow the same approach that made M5 and M6 practical:

- keep the guest-visible contract runtime-owned and explicit
- prefer truthful, inspectable state over hidden provider-side memory
- allow the first durable implementation to be host-backed if needed
- validate behavior through harness artifacts instead of trusting prompt prose

The first M7 implementation may therefore use:

- runtime-owned memory records
- host-persisted session artifacts
- bounded compaction and summarization passes
- host-visible inspection tools and artifacts

This is acceptable as long as:

- remembered facts are traceable
- compaction is truthful
- persistence is bounded and explicit
- resumed tasks reconstruct the same working state the runtime claims to have

## Non-goals

- unbounded conversation retention
- hidden provider-native memory as the primary mechanism
- automatic long-term personal profiling
- speculative fact storage without provenance
- replacing source fetch or workspace inspection with memory guesses
- silent context dropping without auditability

## Core Definition

MiniAgentOS M7 is done when the runtime supports:

- one explicit working-memory model
- one bounded context-compaction path
- one inspectable memory state surface
- one durable resume path for interrupted work
- one harness suite that validates memory truthfulness, compaction, and resume

## Required New Capability Areas

### Memory Classes

M7 should distinguish at least these classes of runtime memory:

- task memory
- source memory
- workspace memory
- execution memory
- recent conversation

These classes should not be flattened into one opaque blob.

### Truthful Compaction

Large tool results and long histories should not be copied forward verbatim by
default. They should be compacted into bounded summaries that preserve:

- actionable facts
- provenance
- unresolved questions
- explicit uncertainty when detail was dropped

Compaction should be a runtime-owned transformation, not an invisible side
effect.

### Context Budgeting

The runtime should know how much context each layer is consuming.

At minimum, M7 should make it possible to inspect:

- instruction size
- current-request size
- memory-layer size
- retained conversation size
- latest-tool-result size

The purpose is not exact token billing. The purpose is controllable prompt
assembly.

### Memory Inspection

Users and the harness should be able to inspect what the runtime currently
believes is important.

This should include:

- current memory entries
- the class of each entry
- where the entry came from
- whether the entry is raw, compacted, or derived

### Resume And Recovery

M7 should let the runtime resume a task after interruption without pretending
the old context still lives only in a transient prompt window.

The resumed runtime should reconstruct its working state from persisted memory,
not from wishful assumptions.

## Prompt And Session Contract

M7 should extend the existing M4/M6 layered prompt contract into a more
explicit memory-aware shape, such as:

- `Current request`
- `Latest tool result`
- `Working memory`
- `Known sources`
- `Workspace memory`
- `Session state`
- `Recent conversation`

The exact labels may evolve, but the important change is structural:

- memory becomes an explicit layer
- not all retained context is conversation
- not all retained context is raw tool output

## Default M7 Surface

The first M7 implementation does not need to expose a large new planner-visible
tool family.

It should, however, provide at least one inspectable memory surface, which may
take the form of:

- shell commands such as `memory-status` or `memory-dump`
- harness artifacts such as `memory_snapshot.json`
- optional runtime tools such as `list_memory` and `read_memory`

The model should not need unrestricted direct write access to memory records in
order for M7 to succeed. By default, compaction and persistence should remain
runtime-governed.

## M6 Compatibility Requirement

M7 must remain additive on top of M6, M5, and M4.

That means:

- M4 session behavior must keep working
- M5 coding loops must keep working
- M6 search and research loops must keep working
- the absence of durable memory configuration must not break ordinary shorter
  tasks

M7 should improve long-task behavior without forcing every request through a
heavy persistence mechanism.

## Harness Bar

The first M7 harness bar should focus on truthful memory behavior, not on
whether the runtime merely stores more text.

Minimum case classes:

- `m7-followup-after-large-tool-result`
- `m7-research-memory-followup`
- `m7-coding-memory-followup`
- `m7-resume-interrupted-task`
- `m7-memory-inspection`
- `m7-truthful-compaction`

The live bar should eventually include at least:

- one QEMU-backed long research follow-up that reuses compacted source memory
- one QEMU-backed coding task that resumes after interruption
- one inspectable memory artifact proving what context was retained

## Success Criteria

M7 is complete when MiniAgentOS can:

1. retain explicit working memory across multiple turns
2. compact oversized context truthfully
3. expose what it currently remembers and why
4. keep research and coding memory separated enough to stay understandable
5. resume interrupted tasks from persisted working state
6. validate all of the above through stable harness artifacts and cases
