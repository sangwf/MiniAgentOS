# Milestone 5: Bounded Coding Substrate

## Status

Implemented.

The repository now has both a harness-first M5 acceptance slice and a
real-runtime M5 acceptance slice. The main regression entrypoints are:

- `./bin/run-suite --suite m5 --config harness/config.fixture.json`
- `./bin/run-suite --suite m5live --config harness/config.runtime-m5.json`

The live suite now proves the real runtime can complete both:

- one bounded `run_process -> read_process_output -> final` loop
- one bounded live `inspect -> edit -> run -> observe -> repair` loop

through the real guest, real host bridge, real Docker-backed process run, and
real OpenAI follow-up.

That is enough to satisfy the core M5 exit bar. Future work can extend the
substrate further, but M5 is no longer partial.

## Objective

Add the first real coding substrate to MiniAgentOS so the system can complete a
bounded inspect/edit/execute/observe loop against a real workspace.

M5 is where MiniAgentOS crosses the line from:

"an agent with network and platform tools"

to:

"an agent that can inspect code, modify files, run a bounded task, observe the
result, and iterate."

## Why M5 Exists

M4 deliberately stayed honest. It gave MiniAgentOS the right loop shape and the
right tool abstractions, but it did **not** pretend the runtime already had:

- a workspace
- file editing
- executable tasks
- process output observation

Those are the minimum capabilities that make coding work real instead of
simulated.

So M5 exists to introduce the minimum substrate needed for the canonical coding
loop:

`inspect -> edit -> execute -> observe -> iterate`

## Product Meaning

By the end of M5, the right mental model should be:

"MiniAgentOS is a bounded coding agent runtime with a real workspace and a real
execution substrate."

This still does not require MiniAgentOS to become a traditional full operating
system or a POSIX clone. It does require the runtime to support the minimum
primitives that make code iteration real rather than simulated.

## Implementation Principle

M5 should prioritize a real contract and a real harness bar over premature
guest-native operating-system breadth.

The first M5 implementation may therefore use:

- a runtime-owned tool contract inside the MiniAgentOS guest
- a host-backed bounded workspace implementation
- a host-backed bounded execution backend

This is acceptable as long as:

- the workspace is real
- file changes are real
- execution is real
- execution output is real
- the agent loop can observe and act on those results

The agent should not know or care whether the substrate is backed by an
in-guest implementation or by a host bridge. It should only see a stable,
bounded runtime-owned tool surface.

## Non-goals

- full POSIX compatibility
- unrestricted shell access
- guest-native general-purpose filesystem parity
- guest-native general-purpose process management
- arbitrary package installation
- multi-user permissions
- full process supervision
- large-scale build farm orchestration

## Core Definition

MiniAgentOS M5 is done when the runtime supports:

- one bounded workspace abstraction
- one bounded file editing tool set
- one bounded execution substrate
- one observable execution output channel
- one coding-oriented harness suite that validates inspect/edit/execute/observe
  loops

## Required New Capability Areas

### Workspace

MiniAgentOS needs a runtime-owned workspace abstraction. This can be minimal,
but it must let the agent reason about and operate on a controlled file tree.

For the first M5 implementation, that workspace may be backed by a host-side
directory tree as long as the runtime enforces:

- one explicit workspace root
- bounded path access beneath that root
- predictable file metadata and file contents
- traceable file operations

### File Editing

The agent must be able to inspect and modify files inside that workspace.

This capability is satisfied only when the runtime can perform real file
operations against a real workspace, not when the model merely emits a textual
patch suggestion without the runtime applying it.

### Execution

The agent must be able to launch at least one bounded executable workflow and
observe the result.

This does **not** require a general-purpose shell first. It does require a real
execution primitive with real output.

The first M5 implementation may back this execution primitive with a host-side
sandbox runner, including a container-based sandbox such as Docker, as long as
execution remains:

- policy-bounded
- trace-visible
- observable by the agent loop

## Core Loop Requirement

M5 keeps the M4 sessioned tool loop, but extends the tool surface so the loop
can now perform coding work.

The important change is not a different loop. The important change is a more
powerful and more truthful substrate.

## M4 Compatibility Requirement

M5 must be an additive milestone on top of M4, not a replacement runtime.

That means:

- the existing M4 sessioned loop must remain intact
- existing M4 tool flows must remain usable
- M5 adds new substrate-backed tools without regressing the existing M4 tool
  surface
- the absence of an M5 workspace or runner backend must not break ordinary M4
  usage

In practice, M4 should remain a valid subset of M5:

- M4-only requests should continue to work without depending on the new coding
  substrate
- M5-specific tools may return structured `unsupported` or `unavailable`
  results when the coding substrate is not configured
- M4 harness expectations should remain stable while M5 introduces new harness
  coverage

## Default M5 Tool Surface

M5 should add at least these tools on top of the M4 set:

- `list_workspace`
- `read_file`
- `write_file`
- `apply_patch`
- `run_process`
- `read_process_output`

Optional later additions:

- `workspace_search`
- `kill_process`
- `list_processes`

### Tool Semantics

- `list_workspace`
  enumerate files and directories within the bounded workspace
- `read_file`
  read one file from the bounded workspace
- `write_file`
  replace or create one file in the bounded workspace
- `apply_patch`
  apply a bounded patch update to one or more files
- `run_process`
  start one bounded execution task
- `read_process_output`
  retrieve output and status for a previously started task

## Honest Execution Requirement

M5 should not hide execution behind fake names like `run_task` if nothing new
is actually happening.

If execution is introduced, it must be real:

- a task starts
- output exists
- the agent can read that output
- the agent can decide what to do next based on it

The runtime does not need to expose a general shell language to satisfy this
bar. A structured execution tool is acceptable and is preferable for the first
M5 implementation if it improves policy clarity and harness stability.

## Policy Requirement

M5 must keep both workspace access and execution bounded.

At minimum this includes:

- workspace root restriction
- file size limits
- patch size limits
- executable allowlist or runner policy
- timeout limits
- output size limits
- concurrent process limits
- optional network policy for sandboxed runs

## Harness Requirement

M5 requires a new coding-oriented harness suite.

The harness should verify:

- the agent can inspect workspace contents
- the agent can read a file and modify it
- the agent can run a bounded process
- the agent can read process output
- the agent can fix a small failing case through one or more edit/run loops

### Minimum M5 Cases

- `m5-read-file-and-answer`
- `m5-edit-file-and-verify`
- `m5-run-process-and-read-output`
- `m5-fix-small-regression`

## Definition Of Done

M5 is complete when all of the following are true:

- MiniAgentOS keeps the M4 sessioned tool loop
- MiniAgentOS preserves ordinary M4 usage without requiring the new M5
  substrate
- the runtime exposes a bounded workspace
- the runtime exposes real file read/write/patch tools
- the runtime exposes a real bounded execution primitive
- process output is observable to the loop
- at least one harness case validates a real inspect/edit/execute/observe cycle

## Exit Bar

If MiniAgentOS still cannot inspect files, modify files, launch a bounded run,
and observe the output, then it is not yet a coding agent runtime.

If MiniAgentOS can complete a small inspect/edit/execute/observe loop inside a
bounded workspace, then the core M5 bar is satisfied.

## Current Runtime Slice

The first runtime-side M5 slice is intentionally narrower than the full M5
target.

It currently exposes these guest shell commands:

- `m5-status`
- `m5-list [path]`
- `m5-read <path>`
- `m5-write <path> <content>`
- `m5-apply-patch <escaped-patch>`
- `m5-run <path>`
- `m5-output <process-id>`

Those commands talk to a host-backed HTTP bridge at `10.0.2.2:8090` and
currently use these bridge routes:

- `GET /healthz`
- `GET /workspace/list`
- `GET /workspace/read`
- `POST /workspace/write`
- `POST /workspace/apply-patch`
- `POST /process/run-python`
- `GET /process/output`

This proves that the guest runtime can inspect a real bounded workspace, modify
real files, launch a real bounded Python run, and read its real output through
an actual host-backed transport without disturbing the existing M4 loop.

The first agent-facing integration has also landed inside the existing session
loop: the OpenAI-backed loop now advertises and parses the first M5 tool set
and can execute bridge-backed M5 tools inside a real turn.

The current live QEMU-backed regression slice is:

- `m5live-run-process-and-read-output`
- `m5live-fix-small-regression`

using:

- `harness/config.runtime-m5.json`
- native guest OpenAI transport with the required host SOCKS5 proxy on `10808`
- boot-time local TLS self-test disabled for harness runs
- host-backed M5 bridge artifacts written into the case output directory

That is enough to prove the real runtime can now:

- receive a live model turn in QEMU
- issue a real M5 tool call from the guest loop
- execute a real bounded host-backed process
- read the resulting stdout back into the loop
- produce a final answer from the observed output
- inspect workspace state before editing
- apply a bounded patch
- rerun a failing check until a passing result is observed

On this host, the supported live OpenAI path for the real-runtime M5 suite is
the native guest transport with the host SOCKS5 proxy on port `10808`. Direct
internet access remains a separate transport research line and is not required
for M5 completion.
