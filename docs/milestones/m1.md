# Milestone 1: Governed Skill Runtime

## Objective

Build a bootable agent runtime that can choose among multiple real skills under
explicit policy and budget constraints, call a real model capability, and be
evaluated automatically across success, refusal, and failure paths.

## Why M1 Exists

Milestone 0 proved one real closed loop:

`boot -> prompt -> one structured task -> fixed skill chain -> external effect`

That was enough to prove MiniAgentOS is real, but not enough to prove it is a
governed agent runtime. In M0, the task kind and execution path are still
mostly fixed in the runtime. M1 exists to prove three stronger claims:

- the runtime can expose more than one planner-visible action
- the runtime can reject actions that violate policy
- the runtime can call a real model capability without breaking harness
  observability

## Non-goals

- multi-agent orchestration
- self-modifying runtime
- general-purpose filesystem
- GUI or browser control
- open-ended internet dependence in automated tests
- Twitter or other social API integrations as a required acceptance path

## Core Definition

MiniAgentOS M1 is done when the runtime supports:

- one agent-first prompt and task entry path
- one structured task contract with explicit policy fields
- one skill registry that maps skill names to handlers
- one policy layer that approves or denies each skill call
- one real model-backed skill
- one harness suite that tests success, refusal, and failure outcomes
- one live OpenAI-backed validation path using `OPENAI_API_KEY` and
  `gpt-5-mini`

## Skill And Tool Model

### Skill

A skill is a planner-visible action with task semantics, policy boundaries, and
trace visibility.

Examples:

- `fetch_url`
- `call_model`
- `post_result`

Skills are the unit of:

- planning
- policy checks
- step budgeting
- harness evaluation
- `TRACE` events

### Tool

A tool is an internal reusable capability used to implement one or more skills.
Tools are not selected directly by the planner in M1.

Examples:

- URL parsing
- DNS resolution
- TCP connect
- TLS handshake
- HTTP request encode/decode
- JSON field extraction

### Required Relationship

The runtime must treat the layers like this:

`goal -> plan -> skill -> tool -> device/network/runtime primitive`

In M1:

- the planner chooses skills, not tools
- policy is enforced at the skill boundary
- traces must always show skills
- traces may optionally show tool-level diagnostics, but tool traces do not
  replace skill traces

## Required Runtime Behavior

The runtime must still boot into `Goal >` and accept one JSON task on stdin,
but the accepted task shape must become more general than the M0 fixed task.

The runtime must:

1. parse a structured task with goal, inputs, allowed skills, allowed hosts,
   and execution budget
2. convert that task into an internal plan made of one or more skills
3. check policy before every skill call
4. execute only allowed skills against allowed hosts
5. emit structured trace events for approval, denial, success, and failure
6. return one structured result payload to the sink

## Default Task Contract

M1 should standardize on one general task form:

```json
{
  "goal_id": "m1-summarize-url",
  "goal": "Fetch the source URL, summarize it in three bullet points, and post the result.",
  "kind": "agent_task",
  "source_url": "http://10.0.2.2:8082/source",
  "sink_url": "http://10.0.2.2:8081/result",
  "model_url": "http://10.0.2.2:8090/summarize",
  "allowed_skills": ["fetch_url", "call_model", "post_result"],
  "allowed_hosts": ["10.0.2.2"],
  "constraints": {
    "max_steps": 6,
    "max_fetch_bytes": 8192
  },
  "output": {
    "format": "bullet_summary",
    "max_items": 3
  }
}
```

The exact field names can still evolve, but M1 must include these concepts:

- goal identity
- goal text
- skill allowlist
- host allowlist
- model endpoint
- step budget
- output shape

## Required Skills

M1 requires these planner-visible skills:

- `fetch_url`
- `call_model`
- `post_result`

These are enough to demonstrate:

- network retrieval
- model invocation
- externally visible side effects

Optional extension skills are allowed, but M1 must not depend on them.

## Required Policy Checks

Before each skill call, the runtime must validate at least:

- the skill name is allowed for this task
- the target host is allowed for this task
- the step budget is not exceeded
- the runtime is not already in a terminal state

The runtime should emit a refusal instead of silently falling through when a
policy check fails.

## Required Trace Events

M1 extends the M0 trace vocabulary. At minimum, the runtime must support:

- `goal_received`
- `intent_parsed`
- `plan_created`
- `policy_checked`
- `skill_called`
- `skill_result`
- `skill_denied`
- `goal_completed`
- `goal_refused`
- `goal_failed`

### Success Path

The success path must include this ordered sequence:

- `goal_received`
- `intent_parsed`
- `plan_created`
- `policy_checked`
- `skill_called`
- `skill_result`
- `goal_completed`

### Refusal Path

The refusal path must include this ordered sequence:

- `goal_received`
- `intent_parsed`
- `plan_created`
- `policy_checked`
- `skill_denied`
- `goal_refused`

### Failure Path

The failure path must include this ordered sequence:

- `goal_received`
- `intent_parsed`
- `plan_created`
- `policy_checked`
- `skill_called`
- `skill_result`
- `goal_failed`

## Real Model Requirement

M1 must replace the M0 local summarizer with one real model-backed path.

The required form for M1 is a host-side model gateway:

- MiniAgentOS sends an HTTP request to a model gateway
- the gateway calls a real model provider
- the gateway returns structured output to MiniAgentOS

This still counts as real model execution because:

- the model call is real
- the network hop is real
- the runtime remains fully observable and testable

### Required Provider For M1

The live model path for M1 must use:

- `OPENAI_API_KEY` for authentication
- `gpt-5-mini` as the model

The OpenAI call may be made by a host-side gateway rather than directly by the
kernel, but the runtime must still issue a real network request to that gateway
and consume the returned summary through the normal skill flow.

### Harness Modes

M1 intentionally keeps two harness modes:

- deterministic mode
  uses a local mock model backend so regressions stay cheap and repeatable
- live mode
  uses a host-side OpenAI-backed gateway with `OPENAI_API_KEY` and
  `gpt-5-mini`

Both modes matter:

- deterministic mode protects day-to-day development
- live mode proves the real model capability works end to end

## Harness Suite

M1 should introduce a suite, not just a single case.

The minimum suite should contain these cases:

### `m1-allow-fetch-model-post`

- allowed host
- allowed skills
- model gateway available
- expected result: `goal_completed`
- must support both deterministic and live OpenAI-backed execution

### `m1-deny-disallowed-host`

- source or sink host is outside the allowlist
- expected result: `goal_refused`

### `m1-deny-disallowed-skill`

- required skill is missing from `allowed_skills`
- expected result: `goal_refused`

### `m1-model-gateway-error`

- model gateway is reachable but returns an error or invalid output
- expected result: `goal_failed`

## Evaluation

The M1 evaluator must check more than M0:

- boot prompt observed
- trace events include the correct ordered path for the case
- step budget not exceeded
- denied skills are not executed
- disallowed hosts are not contacted
- success cases produce a valid sink payload
- refusal cases produce a structured refusal result
- failure cases produce a structured failure result

The repository should support both of these execution styles:

- deterministic suite run against the local mock gateway
- live model validation run against the OpenAI-backed gateway

## Definition Of Done

Milestone 1 is done only when the real MiniAgentOS runtime satisfies all of
the following:

### Runtime

- the runtime still boots in QEMU and enters `Goal >`
- the runtime no longer depends on one hardcoded M0 task kind
- the runtime exposes a real skill registry with at least the required M1
  skills
- the runtime performs policy checks before every skill call

### Real Capabilities

- `fetch_url` performs a real network fetch
- `call_model` performs a real network call to a model gateway
- the live `call_model` path uses `OPENAI_API_KEY` and `gpt-5-mini`
- `post_result` performs a real network POST to the sink

### Observability

- traces include policy decisions as well as skill execution
- success, refusal, and failure each have distinct terminal trace events
- harness artifacts remain diffable under `output/`

### Harness

- the repository contains at least four M1 harness cases
- the M1 suite can be run without manual UART interaction
- the evaluator can mark each case pass or fail without hand inspection
- `./bin/run-suite --config harness/config.example.json` passes as the
  deterministic regression bar
- `./bin/run-case harness/cases/m1-allow-fetch-model-post/task.json --config harness/config.openai.json`
  passes as the live OpenAI validation bar

### Engineering Bar

- M1-specific logic does not continue to grow as one undifferentiated block in
  `runtime/src/main.rs`
- the implementation is split into at least these conceptual areas:
  - task parsing
  - policy checks
  - skill dispatch
  - model request/response handling
- the resulting structure is good enough that a follow-on milestone can add
  skills or policy rules without first rewriting the whole M1 path

### Product Meaning

- M1 proves MiniAgentOS is no longer just a fixed demo loop
- M1 proves MiniAgentOS can make bounded action choices under policy
- M1 proves MiniAgentOS can use a real model capability without giving up
  harness discipline

## Not Done Yet If

Milestone 1 is not done if any of these are still true:

- the planner still only supports one fixed skill chain
- policy is checked only informally or outside the runtime
- the model step is still a local stub with no real model call
- the live model path does not use `OPENAI_API_KEY` and `gpt-5-mini`
- refusal and failure share the same terminal behavior
- the harness can only validate happy-path success
