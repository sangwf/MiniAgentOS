# Milestone 4: Honest Tool Loop

## Objective

Turn MiniAgentOS from an action-branch-driven agent shell into a standard
single-agent `session + model -> tools -> model` runtime, while staying honest
about the capabilities the system actually has today.

M4 is **not** the milestone where MiniAgentOS becomes a full coding agent.
M4 is the milestone where MiniAgentOS adopts the right loop shape and the right
tool abstractions without pretending to already have a workspace or execution
substrate.

## Why M4 Exists

M3 proved MiniAgentOS can:

- accept ordinary natural language at `Goal >`
- use a model in-guest
- compile structured intent
- complete a bounded task and return a result

But M3 still leans too hard on fixed action families and prewired execution
paths.

What M3 does **not** yet provide is the standard agent runtime shape used by
systems such as nanobot and the pi/OpenClaw family:

- one persistent session
- one explicit `model -> tools -> model` loop
- one bounded tool registry
- one clean distinction between user turns, assistant turns, tool calls, and
  tool results

M4 exists to fix the runtime shape first.

## Product Meaning

By the end of M4, the right mental model should be:

"MiniAgentOS is a bootable terminal agent with session state and a bounded
tool loop."

Not:

"MiniAgentOS is already a coding agent."

That stronger claim belongs to the next milestone.

## Non-goals

- full coding-agent parity
- generic shell command execution
- a traditional file system
- a process model
- unrestricted browser automation
- multi-agent orchestration
- long-term memory or retrieval
- self-modifying code flows

## Honest Boundary

M4 must stay honest about what the runtime can and cannot do.

At M4, MiniAgentOS should have:

- session state
- a model loop
- a bounded tool registry
- real network and platform tools
- policy and trace around those tools

At M4, MiniAgentOS should **not** claim to have:

- arbitrary command execution
- file editing over a real workspace
- process management
- a genuine coding workflow

If those things are missing, they should be deferred to M5 rather than hidden
behind names like `run_task` or fake command wrappers.

## Core Definition

MiniAgentOS M4 is done when the runtime supports:

- one primary session-scoped agent loop at `Goal >`
- one bounded tool registry exposed to the model
- repeated tool calls within one turn until stop or budget exhaustion
- multi-turn follow-up within the same session
- direct final answers to the terminal by default
- policy enforcement at the tool boundary
- structured trace for assistant turns, tool calls, tool results, and stop
  reasons
- harness evaluation of loop behavior rather than only one-shot task output

## Architecture Direction

M4 should align the runtime to this shape:

`goal shell -> session manager -> agent loop -> tool registry -> primitives`

### Required Internal Layers

- `goal_shell`
  terminal UX, input editing, status, and final answer rendering
- `session_manager`
  in-memory conversation state and turn history
- `agent_loop`
  the repeated `model -> tool -> model` execution loop
- `tool_registry`
  tool schemas, dispatch, policy hooks, and result normalization
- `model_adapter`
  guest-direct OpenAI request/response handling
- `capability_primitives`
  HTTP, TLS, X integration, trace, and terminal rendering internals

## Core Loop Requirement

The official M4 loop should behave like this:

1. accept a user message into the current session
2. build model context from system prompt, session history, and available tools
3. call the model with tool schemas
4. if the model returns tool calls:
   - validate each call through policy
   - execute the tool
   - append structured tool results to session history
   - continue the loop
5. if the model returns final assistant content:
   - render it to the terminal
   - store it in session history
   - end the turn
6. if the loop exceeds iteration budget or hits an unrecoverable error:
   - return a clear terminal error
   - emit a stop reason in trace

The key design rule is:

**the model decides when to answer and when to call tools.**

## Prompt Contract

M4 must stop treating model input as one undifferentiated string blob.

The guest-owned model input should be built as an explicit layered contract:

1. `Current request`
2. `Latest tool result`
3. `Session state`
4. `Recent conversation`

The critical rule is:

**`Current request` is authoritative.**

That means:

- the latest user turn must remain visible as its own section
- it must not be hidden only inside session history
- old context must never override the current request
- failed or completed prior turns should only matter when the current request
  explicitly asks for follow-up

### Minimum M4 Prompt Semantics

- `Current request`
  the latest user message for the current turn
- `Latest tool result`
  the most recent structured tool output or preview, if any
- `Session state`
  bounded key/value state intended for follow-up behavior
- `Recent conversation`
  the newest suffix of prior user/assistant/tool turns, excluding the current
  user turn when it is already represented in `Current request`

### Bounded Growth Rule

M4 should not try to consume the full model context window. Instead it should
enforce bounded guest-side budgets by section, and when truncation is needed it
should preferentially keep:

1. the full current request
2. the latest tool result
3. session state
4. the most recent suffix of conversation history

This rule exists to prevent the classic bootstrap-agent failure mode where a
new request is silently drowned out by older history or by a long earlier tool
result.

### Current M4 Focus

The next prompt-quality work inside M4 should prefer strengthening the section
boundaries above instead of introducing a heavier request-typing layer. In
practice that means:

- keep `Current request` fully visible whenever possible
- keep `Latest tool result` compact and normalized
- keep `Session state` small and structured
- keep `Recent conversation` as a bounded suffix, not a competing primary
  context source

## Tool Model

M4 should stop treating model invocations as planner-visible tools.

`call_model` belongs to the loop machinery. The public tool surface should
describe external capabilities the loop can invoke.

### Default M4 Tool Surface

The initial M4 tool registry should stay intentionally small and should only
contain tools the runtime can honestly support:

- `fetch_url`
- `post_url`
- `post_tweet`
- `search_recent_posts`
- `get_user_posts`
- `read_session_state`
- `write_session_state`

## Transport Note

The OpenAI-backed M4 loop may reuse a guest-side OpenAI transport across
consecutive OpenAI-only turns. This is an implementation optimization to reduce
repeated TCP, SOCKS5, and TLS handshakes; it does not change the conceptual
`model -> tools -> model` contract.

### Initial Tool Semantics

- `fetch_url`
  fetch one URL and return normalized response text and metadata
- `post_url`
  submit one HTTP POST request to an allowed endpoint
- `post_tweet`
  publish one X/Twitter post through the runtime-owned X integration
- `search_recent_posts`
  search recent X/Twitter posts by query
- `get_user_posts`
  retrieve recent X/Twitter posts for one username
- `read_session_state`
  read bounded per-session state
- `write_session_state`
  write bounded per-session state

These must be real tool definitions with explicit argument schemas, not just
shell commands that happen to exist.

## Session Requirement

The minimum M4 session contract should include:

- one current session at `Goal >`
- prior user, assistant, and tool messages preserved for follow-up turns
- one explicit session reset command
- one explicit new session command
- bounded context growth
- a visible distinction between user turns, assistant turns, and tool results
- failed turns written back into session history so the next turn does not look
  like an unfinished continuation

Persistence across reboot is optional for M4. Stable in-memory session
behavior within one runtime instance is required.

## Trace Requirement

M4 trace should make the loop visible without overwhelming the user-facing UI.

At minimum, trace must expose events equivalent to:

- `session_started`
- `user_turn_received`
- `model_turn_started`
- `model_turn_completed`
- `tool_call_requested`
- `tool_call_denied`
- `tool_call_started`
- `tool_call_completed`
- `assistant_response_rendered`
- `loop_stopped`

For tool results, the harness must be able to distinguish:

- tool name
- arguments
- policy decision
- result status
- whether the loop continued or stopped after the result

## Policy Requirement

M4 keeps policy strict at the tool boundary.

At minimum this includes:

- tool allowlist
- host allowlist / public-host policy
- iteration budget
- session-state key restrictions
- X write/read tool restrictions as needed

## Harness Requirement

M4 requires harness coverage for the loop itself, not just final answers.

The harness should verify:

- the model can call at least one tool and continue the loop
- the runtime can answer a follow-up question using same-session context
- denied tool calls are surfaced clearly
- iteration budgets terminate runaway loops cleanly
- the official live path uses guest-direct OpenAI access
- at least one X reading tool and one X writing tool behave correctly

### Minimum M4 Cases

- `m4-loop-summarize-url`
- `m4-loop-post-url`
- `m4-loop-followup-question`
- `m4-loop-tool-denied`
- `m4-loop-iteration-budget`
- `m4-loop-post-tweet`
- `m4-loop-search-recent-posts`
- `m4-loop-get-user-posts`

## Definition Of Done

M4 is complete when all of the following are true:

- the primary manual path is a sessioned `model -> tools -> model` loop
- the runtime exposes a small bounded tool registry
- one turn may contain multiple tool invocations before the final answer
- the user can ask at least one meaningful follow-up in the same session
- terminal output stays product-like while trace remains harness-visible
- model input uses an explicit layered prompt contract where the current
  request is first-class and authoritative
- the official live path is still guest-direct OpenAI with `OPENAI_API_KEY` and
  `gpt-5-mini`
- the initial X tool set includes both read and write coverage
- harness cases validate loop behavior, not just one-shot action execution

## Exit Bar

If MiniAgentOS still relies on hardcoded action families such as
`local_summary` and `post_summary` to decide the main execution path, then M4 is
not done yet.

If MiniAgentOS instead runs a bounded sessioned tool loop over a real, honest
tool surface, returns answers directly to the shell, and supports follow-up
turns on the same session, then the core M4 bar is satisfied.
