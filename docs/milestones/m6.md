# Milestone 6: Bounded Web Search And Research Runtime

## Status

Partially implemented.

M6 now includes:

- a fixture-backed `m6` harness suite for deterministic search/fetch/follow-up
  validation
- a first live `m6live-search-and-answer` case that exercises the real QEMU
  runtime, real OpenAI model turns, a Brave-backed host search bridge, and a
  real fetch-backed supporting source

M6 is not complete yet. The remaining gap is expanding live research coverage
beyond this first bounded slice.

## Objective

Add the first real search and research substrate to MiniAgentOS so the runtime
can complete a bounded:

`search -> inspect results -> fetch sources -> compare -> summarize -> follow-up`

loop against the open web.

## Why M6 Exists

M4 gave MiniAgentOS a truthful sessioned `model -> tools -> model` loop.
M5 gave that loop a truthful coding substrate. But the runtime still lacks a
real retrieval layer for open-ended research work.

Today MiniAgentOS can:

- fetch one URL
- read one page
- summarize one source

but it cannot yet:

- search for candidate sources
- compare multiple sources
- keep a bounded set of known sources across follow-up turns
- answer research questions without the user already providing the URL

M6 exists to add that missing retrieval layer.

## Product Meaning

By the end of M6, the right mental model should be:

"MiniAgentOS is a bounded research runtime that can search the web, inspect
candidate sources, read selected pages, compare evidence, and answer follow-up
questions from the resulting source set."

M6 does not require MiniAgentOS to become a full browser or a general web
automation system. It does require MiniAgentOS to gain the minimum truthful
primitives needed for multi-source web research.

## Implementation Principle

M6 should follow the same principle that made M5 practical:

- keep the runtime-owned contract inside the guest
- allow the first live implementation to be host-backed
- keep the tool surface bounded, explicit, and policy-controlled
- make the harness a first-class validation layer

The first M6 implementation may therefore use:

- a guest-visible `search_web` tool contract
- a host-backed search bridge
- a live search backend authenticated with `BRAVE_API_KEY`

This is acceptable as long as:

- search results are real
- result metadata is real
- fetched source content is real
- source selection stays visible in trace and artifacts
- follow-up turns can reuse a truthful source set

## Non-goals

- full browser automation
- arbitrary JavaScript page rendering
- unrestricted crawling
- guest-native search indexing
- hidden provider-side answer synthesis as the primary path
- replacing the existing `fetch_url` capability

## Core Definition

MiniAgentOS M6 is done when the runtime supports:

- one bounded web search primitive
- one bounded source-set model inside the session loop
- one research-oriented harness suite for search and follow-up behavior
- one live search path using `BRAVE_API_KEY`
- truthful multi-source research turns through the real runtime

## Required New Capability Areas

### Search

MiniAgentOS needs a runtime-owned search primitive that returns structured web
results for a user query.

For the first M6 implementation, the search backend may be host-backed, but the
guest-visible tool contract must stay stable and provider-agnostic.

### Source Selection

MiniAgentOS must be able to distinguish:

- search-result snippets
- fetched page content
- model-written synthesis

This distinction must remain visible enough for harness evaluation and trace
inspection.

### Research Memory

MiniAgentOS must preserve a bounded set of known sources inside the session so
follow-up turns can build on prior evidence instead of restarting from nothing.

### Multi-source Synthesis

MiniAgentOS must be able to:

- search
- fetch more than one source
- compare sources
- answer from the combined evidence set

M6 is not satisfied by a single search result followed by one blind summary.

## Default M6 Tool Surface

M6 should add at least:

- `search_web`

M6 should continue using these existing capabilities:

- `fetch_url`
- session memory from the M4 loop

Optional later additions:

- `list_sources`
- `read_source`
- `clear_sources`

### Tool Semantics

- `search_web`
  perform one bounded web search and return structured search results
- `fetch_url`
  fetch one selected source URL and return page content or extractable text

The first M6 tool surface should stay intentionally narrow. Search and fetch
should remain separate capabilities:

- `search_web` decides what to read
- `fetch_url` retrieves what was chosen

## Default Live Backend

The first live M6 backend should use Brave Search through `BRAVE_API_KEY`.

Reasons:

- it is a real standalone search API instead of a browser scraping hack
- it preserves a clear `search -> fetch -> synthesize` separation
- it fits the host-backed bridge approach already established by M5

The runtime should not depend on Brave-specific semantics beyond the bridge.
The provider-specific logic should live on the host side.

## Prompt And Session Contract

M6 should keep the M4 prompt layering, but add one research-specific context
layer:

- `Current request`
- `Latest tool result`
- `Known sources`
- `Session state`
- `Recent conversation`

This makes follow-up research behavior first-class instead of accidental.

## M5 Compatibility Requirement

M6 must remain additive on top of M5 and M4.

That means:

- existing M4 requests must keep working
- existing M5 coding requests must keep working
- the absence of an M6 search backend must not break ordinary M4/M5 usage
- M6-specific tools may return structured `unsupported` or `unavailable`
  results when the search substrate is not configured

## Harness Bar

The first M6 harness bar should focus on truthful research behavior, not only
on whether a search API was called.

Minimum case classes:

- `m6-search-and-answer`
- `m6-search-then-followup`
- `m6-compare-two-sources`
- `m6-insufficient-evidence-refusal`

The live bar should eventually include at least:

- one real QEMU-backed `search -> fetch -> summarize` run
- one real QEMU-backed follow-up run that reuses the known source set

The first of those live bars now exists as `m6live-search-and-answer`.

## Success Criteria

M6 is complete when MiniAgentOS can:

1. search the web for candidate sources
2. choose which sources to inspect
3. fetch and read those sources
4. distinguish snippets from fetched evidence
5. answer from multiple sources
6. handle a follow-up question from the same research state
7. do all of the above through a bounded, policy-controlled runtime surface
