# Milestone 0: Boot to Live Agent

## Objective

Build a bootable agent-first runtime that can complete one real networked task
with full traceability.

## Non-goals

- general purpose shell
- POSIX compatibility
- full filesystem
- multi-agent orchestration
- self-modifying runtime
- GUI or browser control

## Required loop

1. The system boots into an agent prompt.
2. The harness submits one structured JSON task.
3. The runtime parses the task into an internal intent.
4. The runtime creates a small plan.
5. The runtime executes one or more native skills.
6. The runtime emits trace events during execution.
7. The runtime performs a real external effect.
8. The harness evaluates the run automatically.

## Default case

The default M0 case is `m0-fetch-summarize-post`.

Input:

- fetch a source document from a host-side HTTP fixture
- summarize it
- POST the result to a host-side sink

Required native skills:

- `fetch_url`
- `summarize_text`
- `post_result`

## Harness contract

The runtime must:

- print a line containing `Goal >`
- accept one JSON task on stdin
- emit trace lines prefixed with `TRACE `
- POST a JSON result to the sink URL from the task payload

## Trace events

At minimum the default case expects these events in order:

- `goal_received`
- `intent_parsed`
- `plan_created`
- `skill_called`
- `skill_result`
- `goal_completed`

## External effect

For M0, the external effect is a successful HTTP POST to the harness sink with:

- `goal_id`
- `status`
- `summary`

## Evaluation

The run passes when:

- boot prompt appears before timeout
- sink receives one valid JSON payload
- required trace events appear in order
- skill call count stays within budget
- summary is non-empty and contains required anchor terms from the case

## Definition of Done

Milestone 0 is done only when the real MiniAgentOS runtime, not the fixture
agent, satisfies all of the following:

### Runtime behavior

- QEMU boots into an agent-first prompt that includes `Goal >`.
- The runtime accepts one structured task over stdin without manual editing or
  hardcoded case data.
- The runtime turns that task into an internal execution plan.
- The runtime executes the default M0 skill loop:
  `fetch_url -> summarize_text -> post_result`.

### Real actions

- `fetch_url` performs a real HTTP GET to the source fixture.
- `post_result` performs a real HTTP POST to the result sink.
- The result payload includes `goal_id`, `status`, and `summary`.
- The summary is produced by MiniAgentOS logic, not by the host harness.

### Observability

- The runtime emits machine-readable `TRACE {json}` lines during execution.
- The emitted trace includes the required ordered event sequence.
- The harness captures UART output, parsed trace events, and the sink result as
  run artifacts.

### Harness integration

- `./bin/run-case harness/cases/m0-fetch-summarize-post/task.json --config <real-config>`
  passes against the real MiniAgentOS boot command.
- The evaluator returns `pass: true` without manual inspection.
- The run stays within the configured timeout and skill-call budget.

### Engineering bar

- The M0 path is documented well enough that another person can run it from a
  clean checkout.
- The runtime path used for M0 does not depend on ad hoc terminal interaction
  after boot.
- The implementation does not require a traditional shell, filesystem, or
  general purpose OS features beyond what M0 needs.

## Not Done Yet If

Milestone 0 is not done if any of these are still true:

- only the host-side fixture agent passes the harness
- the system still expects manual UART commands like raw URLs instead of a
  structured task
- trace output is human-readable only and cannot be parsed by the evaluator
- the sink POST is mocked, skipped, or performed by host-side glue instead of
  MiniAgentOS
- success depends on hand inspection rather than `report.json`
