# Milestone 2: Native Goal Shell

## Objective

Build a bootable goal shell that accepts one constrained natural language goal
directly at `Goal >`, compiles it locally inside MiniAgentOS into the governed
task contract from M1, and executes it through the existing skill runtime
without depending on a host-side translation service.

## Why M2 Exists

Milestone 1 proved MiniAgentOS can execute a governed skill runtime under
policy with real network and model calls, but the human-facing entry path is
still machine-shaped:

- JSON task lines for automation
- raw URLs and debug commands for manual use

That is enough for harness-first development, but not enough for MiniAgentOS to
feel like an agent-first runtime when a human types at `Goal >`.

M2 exists to prove three stronger claims:

- a person can type a supported goal directly at `Goal >`
- MiniAgentOS itself can compile that goal into a bounded internal task
- the compiled task still runs through the governed M1 runtime

The important boundary is this:

the harness may drive and evaluate the goal shell, but it must not be the place
where goal understanding fundamentally lives.

## Non-goals

- open-ended chat assistant behavior
- multi-turn dialogue or conversation memory
- replacing the structured JSON task path used by the harness
- browser control or desktop automation
- autonomous background planning loops
- host-side translation services as a required dependency

## Core Definition

MiniAgentOS M2 is done when the runtime supports:

- one plain-text goal path at `Goal >`
- one local compilation stage that turns supported goal text into a structured
  internal task
- one preserved JSON task path for harness and low-level testing
- one governed execution path that reuses the M1 skill and policy runtime
- one deterministic natural-language regression suite
- one manual QEMU interaction path where a supported goal succeeds without any
  special host-side translator

## Interface Model

M2 should treat the shell as a single-port goal interface:

- if input starts with `{`, treat it as a structured task
- otherwise, treat it as a user goal candidate

Legacy debug commands may remain for development, but they are not the primary
interface and are not the measure of M2 completion.

The critical user-facing bars are:

- `Goal > Fetch ... summarize ...`
- `Goal > Fetch ... summarize ... post ...`

Supported goals must succeed without falling back to `unknown command` and
without requiring a host-side goal translation fixture.

## Goal Compilation Model

M2 introduces a local stage ahead of the M1 runtime:

`goal text -> local goal compiler -> structured task -> plan -> skill -> tool`

For M2, that compiler may be deterministic and deliberately narrow. It does not
need to understand arbitrary requests. In fact, it should not pretend to.

### Required Output Shape

The local compiler must produce at least:

- one generated or preserved `goal_id`
- one recognized task kind
- one explicit set of requested action parameters such as URLs and output shape
- one task form that the governed M1 runtime can execute without a separate ad
  hoc path

### Policy Boundary

Natural language must not be allowed to smuggle policy.

That means:

- goal text may request an action
- goal text may include target resources such as URLs
- policy still comes from the runtime or harness profile, not from the user text

The compiler can infer requested actions, but the existing policy layer must
still approve or deny each skill call.

## Required Runtime Behavior

The runtime must:

1. boot into `Goal >`
2. accept one plain-text goal line
3. start a local compilation stage instead of returning `unknown command`
4. compile supported goals into one of the existing bounded task forms
5. execute the compiled task through the local fetch/summary path or the
   governed M1 skill runtime
6. preserve the structured JSON task path for automation and regression
7. return one structured terminal result

## Default Goal Families

M2 should standardize on a tiny set of constrained natural language families,
rather
than pretending to support arbitrary requests.

The default goal families are:

- fetch one source URL and summarize it locally back to the shell
- fetch one source URL, summarize it with the existing model capability, and
  post the result to one sink URL

Examples:

```text
Fetch http://10.0.2.2:8082/source and summarize it in three bullet points.
```

```text
Fetch http://10.0.2.2:8082/source, summarize it in three bullet points, and post the result to http://10.0.2.2:8081/result.
```

This is intentionally narrow. M2 is about proving that natural language can
enter the system natively, not about building a general chat surface. The local
summary path is the primary manual UX bar because it does not require a sink or
model gateway.

## Required Trace Events

M2 extends the M1 trace vocabulary. At minimum, the runtime must support:

- `goal_received`
- `goal_text_received`
- `goal_compilation_started`
- `goal_compiled`
- `goal_compilation_failed`
- `plan_created`
- `policy_checked`
- `skill_called`
- `skill_result`
- `goal_completed`
- `goal_refused`
- `goal_failed`

### Success Path

The success path must include this ordered sequence:

- `goal_received`
- `goal_text_received`
- `goal_compilation_started`
- `goal_compiled`
- `plan_created`
- `policy_checked`
- `skill_called`
- `skill_result`
- `goal_completed`

### Unsupported Path

If the input goal is unsupported, the path must include:

- `goal_received`
- `goal_text_received`
- `goal_compilation_started`
- `goal_compilation_failed`
- `goal_refused`

### Execution Failure Path

If compilation succeeds but execution fails, the path must include:

- `goal_received`
- `goal_text_received`
- `goal_compilation_started`
- `goal_compiled`
- `plan_created`
- `policy_checked`
- `skill_called`
- `skill_result`
- `goal_failed`

## Harness Suite

M2 should add a dedicated natural-language suite.

The minimum suite should contain these cases:

### `m2-nl-fetch-summarize`

- plain-text goal
- valid source URL
- expected result: `goal_completed`
- summary is printed locally to UART with no sink or model requests

### `m2-nl-fetch-model-post`

- plain-text goal
- valid source and sink URLs
- expected result: `goal_completed`

### `m2-nl-refuse-disallowed-host`

- plain-text goal includes a disallowed host
- compilation succeeds
- policy refusal happens after compilation
- expected result: `goal_refused`

### `m2-nl-unsupported-goal`

- plain-text goal asks for something outside the supported goal family
- expected result: `goal_refused`
- the runtime must not silently reinterpret the goal as some other action

### `m2-nl-compilation-error`

- plain-text goal is structurally ambiguous or invalid for the local compiler
- expected result: `goal_refused` or `goal_failed`, but the compilation failure
  must be explicit in trace and result reason

## Evaluation

The M2 evaluator must check more than M1:

- boot prompt observed
- supported natural-language goal does not fall through to `unknown command`
- compilation events appear in the correct order
- compiled execution still obeys policy and step budget
- refusal due to unsupported input is distinguishable from execution failure
- local summary success prints a visible summary back to UART
- governed success cases produce a valid sink payload
- structured JSON task input still works after the natural-language path lands

The harness should preserve enough artifacts to inspect both shell-level and
execution-level behavior. At minimum that includes:

- UART log
- parsed trace
- sink result when the goal shape includes posting

## Definition Of Done

Milestone 2 is done only when the real MiniAgentOS runtime satisfies all of the
following:

### Runtime

- the runtime still boots in QEMU and enters `Goal >`
- the runtime accepts at least one supported plain-text goal form
- supported plain-text goals no longer return `unknown command`
- the runtime still accepts the structured JSON task path from M1
- compiled goals execute through the local fetch/summary path or the governed
  M1 runtime instead of a special ad hoc fast path

### Native Goal Handling

- supported goal compilation happens inside MiniAgentOS
- the supported goal path works in manual QEMU interaction
- the M2 success path does not require a host-side translation service

### Real Capabilities

- the local natural-language success case performs a real fetch and local
  summary with no sink or model dependency
- the governed natural-language success case performs a real fetch, model call,
  and sink POST
- the M1 live summarization path continues to use the real model-backed flow

### Observability

- trace output makes the compilation stage visible
- compilation refusal is distinguishable from policy refusal and execution
  failure
- harness artifacts remain diffable under `output/`

### Harness

- the repository contains at least five M2 natural-language cases
- the M2 suite can be run without manual UART interaction
- one deterministic M2 suite passes as the regression bar
- the M1 JSON-task suite still passes after the natural-language path lands

### Engineering Bar

- M2 goal-shell logic does not get folded back into one large undifferentiated
  UART handler
- the implementation is split into at least these conceptual areas:
  - shell input classification
  - local goal compilation
  - compiled task generation
  - handoff into the governed runtime
- adding a new supported goal family does not require rewriting the skill
  executor

### Product Meaning

- M2 proves MiniAgentOS can be used through a human-facing goal shell
- M2 proves natural language can enter the system without bypassing policy
- M2 proves the shell is native to the runtime rather than delegated to the
  harness

## Not Done Yet If

Milestone 2 is not done if any of these are still true:

- supported natural-language goals still end in `unknown command`
- the natural-language path bypasses the M1 policy layer
- only structured JSON tasks are realistically usable
- goal understanding still depends on a host-side translation service
- unsupported goals are handled by accidental fallthrough instead of structured
  refusal
- the harness cannot validate the natural-language path automatically
